Cruiser Command

Cruiser Command is a multiplayer strategy game created as a custom mod for Starcraft 2. It's a highly tactical game requiring a lot of teamwork to come out victorious over the other team.

I started working on it in 2011 using Starcraft 2's Galaxy Editor. Since then I have kept developing and maintaining it while communicating with the active playerbase about their thoughts and ideas.

Since it went public it has been played by thousands of players and maintained an average rating of over 4.7/5 from over 2000 reviews. The game has a loyal fanbase with some players having played the game for thousands of hours. There's also an active forum with over 150 registered users.

A user-made video compilation can be found here

Cruiser Command has been my by far largest personal project for the last couple of years. It currently contains around 25,000 lines of code, 95% of it written by me alone. The community around the game have also helped out by contributing the loading screen and the interior design of the spaceships.

Over the years of development, I've improved a lot as a developer. I've dealt with all kind of weird bugs, the easily reproducible ones and the ones that only show up once in a blue moon. Developing a game as complex as Cruiser Command means I got the chance to try out all kinds of different programming.

In addition to using modularized code for different parts of the spaceship utilities, I also dealt with advanced Artificial Intelligence. The A.I provided bots to perform a multitude of different tasks in the game. Many were even performed better than the best players could accomplish (which ended up having the unintended effect of experienced players preferring A.I players on their team to inexperienced players).

I also worked on some interesting math problems. For example an A.I weapon aiming prediction to fire weapons that hit the enemy based on the direction and speed they are going. Because of the map layout (basically a torus projected out on a rectangle), I had to figure out a closest distance formula to make the A.I travel around the map efficiently.

Unfortunately, the way Starcraft 2 mods work, it's locked inside a binary file and is not available in human-readable format unless you open it in the editor. For the sake of showing some source code, I've decided to upload the full extracted source code here. Be aware that it's 25,000 rows of raw code without syntax highlighting.

SuperWebRecorder

During the spring of 2014 I spent several months working with 8 other KTH students on a project for the company Beta Family. Beta Family works with offering crowdsourced beta testing for companies by connecting a community of beta testers with a community of developers. To simplify this, we were tasked to create an application that records users' website experiences. This meant recording everything that happened on the site, following the user's path throughout the different pages of the site and recording the interactions. The recordings were then saved in a database to be replayed later on by the developer. It also included aggregation functions to combine replay data for visualizations such as heatmaps.

I was the project manager, but I also spent an active part on the programming group. It was a great experience and my first taste of programming in a slightly larger team. In the end, Beta Family were very happy with the results and so were we.

The project gave me experience in a lot of different web development areas. Since I was the project manager in addition to being part of the programming team, I went around and worked on most parts of the project and helped out when people got stuck. The project consisted of several different parts. There were the recording script recording everything the user does at the site and keeps track of all DOM changes. Everything that happens is serialized and sent as a JSON object to the back-end which handles it and inserts it properly in the database.

There's also the replay feature which fetches recordings from the database and replays them for the developer in control of the site with the recording. I spent most time on this part and, amongst other things, worked on making fully functional time controls to allow you to skip to any point of time in the replay.

There's the backend part with a MySQL database. For development reasons, we maintained support for both a MySQL and PostgreSQL database through a wrapper. We also performed a decent amount of penetration testing to root out any problems with SQL injections or stored XSS injections.

I spent a decent amount of time working on the front-end as well which was made in a combination of Zurb Foundation and AngularJS. I took the non-functional UI design made by another team made and connected it to the back-end and redesigned it as necessary.

Since this project was made for Beta Family, the code is not open source. However, you can see the result at their demo page.

Kartobot

During the spring of 2012, I started working on a robot project with two friends. The project was done as our final high school project and was to be done before summer graduation. One of my friends and I had already been working on a previous project based on building and programming a quadcopter from scratch. Since we were already interested on building and working on robots, we decided to make a new one for our project.

The kartobot travels around on the ground, equipped with an ultrasonic tower. The tower continually rotates back and forth with two opposite ultrasonic sensors on the top. This tower supplies continuous sensor information in regards to the robots distance to nearby objects. Since the tower is rotating, it is possible to draw a simple map of the surroundings. In addition to this, the robot keeps track of the map it draws and aims to explore it by automatically travelling around and avoiding new obstacles that it finds using a pathfinding algorithm.

In the project I was mainly responsible for the tower and the pathfinding while the others focused on tasks like the map drawing, the wireless communication to the computer and the navigation.

Building and programming the tower rotation was relatively simple. I then tested out the margin of error to see what kind of measurements we could expect to get. We found that the ultrasonic sensors send out a pulse in a triangle in front of it instead of just straight forward, so sometimes the results was a bit off.

For the pathfinding I implemented A* pathfinding algorithm that I optimized for the requirements of very low memory that we had. Using some smart compressing and a binary heap implementation, I managed to cut down memory usage to an acceptable level while at the same time making it hundred times faster.

The full source is available in my github repository.

Zorkensburg

Zorkensburg is a text-based adventure game written by me and a friend as a final project in a programming course in C++. The game is mostly an engine and contains a small worldfile as an example of the possibilities it gives. The example game is heavily inspired by the start of Zork, since it gives a lot of opportunities to showcase cool features.

The project was developed with the idea of making it easy to design new games for it without hard-coding doors and locks and other functionality. This idea grew into a complex and powerful scripting language. The scripting functionality is saved in the worldfile and can be saved and loaded to if you want to save your progress. The scripting allows you to implement all kinds of features like: lamp buttons (where you can't see the contents of the room if the light is turned off), Monster AI, traps that occur the first time that player enters a room and special sequences of actions to get to the next room (For example, pulling a rug to reveal a trap-door beneath it and then opening the trapdoor before being allowed to descend to the basement)

The scripting system is based on flags and can save states, some of which have special properties (items with the "lightsource" flag will light up rooms. The system is based on vectors of anonymous functions as well as a map to function pointers. It's made to be easily extendable and the scripting language is well documented over in my github repository.
